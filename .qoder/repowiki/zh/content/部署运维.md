## 简介
本指南面向生产环境，提供 IPv6 代理池在不同环境（物理服务器、虚拟机、容器）下的部署方案，覆盖自动化脚本使用、systemd 服务管理、日志与监控告警、安全加固、性能调优与故障恢复等。项目为纯 Go 实现，支持 HTTP CONNECT 与 SOCKS5 双协议，强制通过 IPv6 出口，内置并发限流，适用于与 v2ray/xray 配合的场景。

**重要更新** 采用 /112 子网隔离最佳实践，确保同一 /64 地址段下多台服务器互不干扰，支持大规模集群部署。

## 核心组件
- 主程序 main.go
  - 启动 HTTP CONNECT 与 SOCKS5 监听端口，解析命令行参数，初始化 IPv6 前缀与并发信号量。
  - **重要更新** 默认前缀从 /64 更新为 /112，支持多服务器部署场景

## 详细组件分析

### 主程序 main.go
- 启动与参数
  - 监听 HTTP CONNECT 与 SOCKS5 端口，默认分别位于 53420 与 53421；可通过 -http/-socks 调整。
  - **-prefix 指定 IPv6 前缀（如 /112），用于随机源地址分配。**
  - -c 限制最大并发连接数，防止资源耗尽。
  - -v 参数启用调试模式，输出详细的 DEBUG 级别日志。
- 日志系统架构
  - 初始化四个独立的日志记录器：infoLog、warnLog、errorLog、debugLog
  - INFO 日志：正常业务流程和重要事件
  - WARN 日志：警告信息和潜在问题
  - ERROR 日志：错误信息和异常处理
  - DEBUG 日志：仅在调试模式下输出的详细调试信息
- 随机源地址生成
  - 根据前缀掩码计算主机位长度，按字节填充随机值，确保在同一 /64 下不同实例互不冲突。
  - **重要更新** 支持 /112 子网，每个服务器拥有独立的 256 个可用地址空间
- HTTP CONNECT 处理
  - 仅允许 CONNECT 方法；若未携带端口则补全默认 443。
  - 使用 Hijack 接管连接，清空残留缓冲，设置 TCP_NODELAY。
  - 通过 net.Dialer 指定 LocalAddr 为随机 IPv6 源地址，超时 30s。
  - 详细的错误日志，包含目标地址、源IP、错误信息等完整上下文。
  - 成功后返回 200，随后双向拷贝数据。
- SOCKS5 处理
  - 握手阶段仅接受无认证模式。
  - 解析请求，支持域名与 IPv6 地址类型，拒绝 IPv4。
  - 同样通过随机 IPv6 源地址拨号，成功后返回 0x00 并转发数据。
  - 完整的错误处理链路，每个步骤都有相应的错误日志。
- 并发控制
  - 使用带缓冲 channel 作为信号量，超出限制直接拒绝，避免内存与文件描述符耗尽。
  - 并发限制的详细日志记录，便于监控和分析。

### 安装脚本 install.sh
- 功能概览
  - 检查 root 权限，安装 golang-go、git、ndppd、curl 等依赖。
  - 创建安装目录 /opt/ipv6-proxy 与配置目录 /etc/ipv6-proxy。
  - 从 GitHub 克隆或下载源码，执行 go mod tidy 与 go build 生成二进制。
  - 写入内核参数至 /etc/sysctl.d/99-ipv6-proxy.conf 并立即生效。
  - 复制 systemd 服务单元到 /etc/systemd/system/ipv6-proxy.service，重载 daemon。
- **重要更新** 注意事项
  - 需要公网 IPv6 前缀并已在本机添加 local 路由。
  - **推荐使用 /112 子网进行多服务器部署，避免地址冲突**
  - 需配置 ndppd 将前缀静态通告给上游路由器。
  - 建议在生产环境固定 Go 版本与构建产物校验。

### 证书生成脚本 gen_cert.sh
- 功能概览
  - 生成 RSA 2048 自签证书，CN 默认取本机外网 IPv4（可通过 ifconfig.me 获取），也可传入自定义 CN。
  - 输出 cert.pem 与 key.pem 到 /usr/local/etc/xray，key.pem 权限设为 600。
  - 提供验证命令输出 Subject 与 SAN 信息。
- 适用场景
  - 与 v2ray/xray 搭配使用时，作为 TLS 证书的快速生成工具。
- 安全建议
  - 生产环境建议使用受信任 CA 签发的证书；自签证书仅用于测试或内网隔离场景。

## 性能与容量规划
- 并发与限流
  - 通过 -c 控制最大并发连接数，建议根据 CPU 核数、内存与目标带宽评估。
  - 观察日志中 "[WARN] 连接数超限" 频率，适当提高 -c 或扩容实例。
- 内核参数
  - 启用 net.ipv6.ip_nonlocal_bind、net.ipv6.conf.all.forwarding。
  - 调整 IPv6 邻居表阈值 gc_thresh1/2/3，避免在高并发下丢包。
  - 开启 net.ipv4.tcp_tw_reuse，缩短 tcp_fin_timeout，扩大 ip_local_port_range。
- 连接复用
  - 与 v2ray/xray 配合使用 smux，减少短连接数量，降低 conntrack 压力。
- 资源监控
  - 监控指标：CPU、内存、FD 使用数、连接数、错误率、延迟分位数。
  - 推荐接入 Prometheus + Grafana，暴露进程级指标或通过 sidecar 采集。

## 故障排查指南
- 日志系统使用
  - 调试模式：使用 `-v` 参数启动程序，启用详细的 DEBUG 级别日志输出
  - 日志级别：
    - `[INFO]`：正常业务流程，如连接建立、请求处理
    - `[WARN]`：警告信息，如方法不支持、连接数超限
    - `[ERROR]`：错误信息，如连接失败、Hijack 失败
    - `[DEBUG]`：详细调试信息，仅在 `-v` 模式下输出
  - 日志过滤：使用 `journalctl -u ipv6-proxy -f --grep "\[ERROR\]"` 过滤错误日志
- 无法绑定端口
  - 现象：启动时报端口占用或权限不足。
  - 排查：确认端口未被占用、非特权端口无需 root；必要时使用 iptables/nftables 放行。
  - 日志定位：查看 `[ERROR]` 级别的启动失败日志
- IPv6 前缀不可达
  - 现象：拨号失败或目标不可达。
  - 排查：检查本机 local 路由是否正确、ndppd 是否运行且规则匹配、上游路由器是否学习到前缀。
  - **重要更新** 确认使用的是 /112 子网而非 /64，避免多服务器地址冲突
  - 日志定位：关注 `[HTTP-FAIL]` 和 `[SOCKS5-FAIL]` 日志，查看具体的错误信息
- 并发受限频繁拒绝
  - 现象：大量 "[WARN] 连接数超限" 日志。
  - 排查：提升 -c 或横向扩容实例；检查下游目标响应时间与错误率。
  - 调试模式：使用 `-v` 参数查看详细的并发控制日志
- 证书问题
  - 现象：TLS 握手失败或 SAN 不匹配。
  - 排查：使用 gen_cert.sh 重新生成，确认 CN/SAN 与实际域名或 IP 一致；生产环境改用受信任 CA。

## 结论
本项目以极简架构实现了高可用的 IPv6 出口代理池，具备双协议支持与并发限流能力。**采用 /112 子网隔离最佳实践**显著提升了多服务器部署的稳定性和可扩展性。通过一键安装脚本与 systemd 服务管理，可在物理机、虚拟机与容器中快速部署。生产环境应重点关注 IPv6 路由与邻居通告、内核参数调优、证书管理与监控告警，并结合 smux 等技术提升整体吞吐与稳定性。

### IPv6 子网隔离最佳实践

**重要更新** 采用 /112 子网隔离方案，确保多服务器部署时的地址空间独立：

#### 多服务器部署示例
```bash
# 服务器A - 使用 240e:6b0:50::/112 子网
sudo ip route add local 240e:6b0:50::/112 dev enp2s0
./ipv6-proxy -prefix 240e:6b0:50::/112

# 服务器B - 使用 240e:6b0:50:2::/112 子网  
sudo ip route add local 240e:6b0:50:2::/112 dev enp2s0
./ipv6-proxy -prefix 240e:6b0:50:2::/112

# 服务器C - 使用 240e:6b0:50:4::/112 子网
sudo ip route add local 240e:6b0:50:4::/112 dev enp2s0
./ipv6-proxy -prefix 240e:6b0:50:4::/112
```

#### NDPPD 配置示例
```bash
# 为每个 /112 子网配置静态邻居通告
sudo tee /etc/ndppd.conf <<'EOF'
route-ttl 30000

proxy enp2s0 {
    router no
    timeout 500
    ttl 30000

    # 服务器A的 /112 子网
    rule 240e:6b0:50::/112 {
        static
    }
    
    # 服务器B的 /112 子网
    rule 240e:6b0:50:2::/112 {
        static
    }
    
    # 服务器C的 /112 子网  
    rule 240e:6b0:50:4::/112 {
        static
    }
}
EOF

sudo systemctl enable ndppd
sudo systemctl restart ndppd
```

#### /112 子网优势
- **地址空间充足**：每个 /112 子网提供 256 个可用 IPv6 地址
- **完全隔离**：不同服务器的地址空间互不重叠
- **易于管理**：清晰的地址分配方案，便于故障排查
- **扩展性强**：支持大规模集群部署，地址规划简单
